<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| trainer client → Firestore | A trainer's browser writes workout_sessions / workout_assignments docs; the trainerId field is attacker-controllable input |
| member client → Firestore | A member's browser reads workout_sessions / workout_assignments; could attempt to read another gym's data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing | workout_sessions create | mitigate | Rule requires `get(trainers/$(trainerId)).data.uid == uid()` — a trainer cannot write a session under another trainer's id |
| T-07-02 | Tampering | workout_assignments create | mitigate | Same cross-doc uid check on trainerId; only owner or the matching trainer can create |
| T-07-03 | Information Disclosure | cross-gym read | mitigate | `allow read: if sameGym()` scopes reads to the caller's gymId; client list() also filters by gymId |
| T-07-04 | Elevation of Privilege | member writing sessions | mitigate | create rule only permits `owner()` or `myRole() == "trainer"`; members have no create path |
| T-07-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan — zero new dependencies; project is build-stepless vanilla JS |
</threat_model>
