# Phase 7 Research: Trainer-Member Workout Assignment

**Researched:** 2026-06-21
**Domain:** Vanilla JS ES Module SPA — Firestore collections, role-based module rendering
**Confidence:** HIGH (all findings from direct codebase inspection)

## Summary

Phase 7 adds three role-specific views on top of a fully established module system.
The codebase has zero build tooling — it is vanilla JS ES modules served directly, with every screen
following an identical `{ render(context), bind(root, context) }` contract. All Firestore access goes
through a single `services.data` abstraction that already supports every needed operation
(list, save, remove). The `workout_assignments` collection is already wired into the data loading
pipeline (`collectionNames` in app.js, `COLLECTIONS` in firebase-init.js, `collections.assignments`
in utils.js) but has no UI. The `workout_sessions` collection is completely new and needs to be
registered in all four places (app.js, firebase-init.js, utils.js, firestore.rules).

Phase 7 requires three new module files (or significant additions to two existing ones), two new nav
entries (one trainer-scoped, one member-scoped), Firestore rules for trainer write access and member
read access to the two workout collections, and smoke-test coverage for the new modules.

**Primary recommendation:** Create two new module files — `modules/trainer-members.js` and
`modules/my-workout.js` — rather than extending existing ones. This avoids bloating
`trainer-checkin.js` and `my-membership.js` with unrelated concerns. The owner view is a small
read-only addition to `trainers.js` (a new section below the existing card grid).

## Existing Patterns

### Module Structure

Every module in `modules/` exports a single object with two methods:

```js
export const myModule = {
  // Returns an HTML string. No DOM access. Called synchronously.
  render(context) {
    const { data, profile, settings, myMember, myTrainer } = context;
    return `<html string>`;
  },

  // Optional. Wires events after render() output is in the DOM.
  bind(root, context) {
    root.querySelector("#form").addEventListener("submit", async (e) => {
      e.preventDefault();
      const saved = await context.services.data.save("collection_name", payload);
      context.toast("Saved.");
      context.applyChange("collection_name", saved);
    });
  }
};
```

Key constraints observed from inspecting all modules:
- `render()` must return an HTML string — no DOM access inside render.
- `bind()` receives the already-mounted DOM root (the `#view` element) and the context.
- `bind()` is optional — modules without interactivity omit it entirely (e.g. `my-membership.js`).
- Role branching happens inside render/bind with `context.profile?.role === "trainer"` checks. Good
  examples: `progress.js` (owner vs. member render paths), `dashboard.js` (owner/member/trainer
  paths).
- The `context.myTrainer` field is the trainer's own roster doc (from `data.trainers` matched by
  `uid`). The `context.myMember` field is the member's own roster doc matched the same way.
- `withButtonLoading(button, asyncFn)` in utils.js handles button disabled-state and spinner during
  async operations — use this for all form submits and action buttons.
- `confirmDialog({ title, body, confirmText, danger })` in utils.js renders an accessible modal for
  delete/destructive confirmations.

### firebase-init.js API

The `services.data` object exposes:

| Method | Signature | Notes |
|--------|-----------|-------|
| `list(collectionName)` | `async list(name) => docs[]` | Filters by gymId automatically. Sorted by updatedAt desc. |
| `save(collectionName, docData)` | `async save(name, data) => savedDoc` | Upserts. Auto-stamps gymId, createdAt, updatedAt. If `docData.id` exists, updates that doc. |
| `remove(collectionName, id)` | `async remove(name, id)` | Deletes by id. |
| `getSettings()` | `async getSettings() => settings` | Returns gym_settings doc. |
| `saveSettings(settings)` | `async saveSettings(s) => s` | Updates gym_settings. |

**gymId scoping:** Both Firebase and local modes automatically inject `gymId` from the authenticated
user's profile. All `list()` calls filter by `gymId`. Callers never need to pass gymId explicitly.

**ID generation:** `save()` auto-generates IDs using `createId(collectionName)` which produces
`"workout_sessions_<8-char-uuid>"` style IDs. Caller can also pass an existing `id` to update.

To add a new collection to local/demo mode: add the collection name string to `COLLECTIONS` array at
the top of `lib/firebase-init.js` (line 3–15). This ensures the local state initializer creates an
empty array for it.

### Firestore Rules

Current rules structure (firestore.rules):

- Helper functions: `signedIn()`, `uid()`, `userDoc()`, `myGymId()`, `myRole()`, `owner()`,
  `sameGym()`, `incomingSameGym()`
- Specific match blocks for: `gym_codes`, `users`, `gym_settings`, `members`, `attendance`,
  `trainers`, `trainer_attendance`
- **Catch-all at line 149:** `match /{collection}/{docId}` — owner read/write, same-gym read.
  This currently covers `workout_templates`, `workout_assignments`, `progress_records`, `reminders`,
  `membership_plans`, `payments`.

**Phase 7 needs explicit rules for two new collections:**

`workout_assignments` — currently falls under the catch-all (owner write only). Phase 7 needs
trainers to write their own assignments.

`workout_sessions` — new collection, not in the rules at all. Needs:
- Trainer can create sessions for their assigned members
- Members can read their own sessions
- Owner can read all

The correct pattern (modeled on the `attendance` rule which uses a cross-doc lookup):
```
match /workout_sessions/{docId} {
  allow read: if sameGym();
  allow create: if incomingSameGym() && (
    owner()
    || (
      myRole() == "trainer"
      && request.resource.data.gymId == myGymId()
      && exists(/databases/$(database)/documents/trainers/$(request.resource.data.trainerId))
      && get(/databases/$(database)/documents/trainers/$(request.resource.data.trainerId)).data.uid == uid()
    )
  );
  allow update, delete: if owner() && sameGym();
}

match /workout_assignments/{docId} {
  allow read: if sameGym();
  allow create: if incomingSameGym() && (
    owner()
    || (
      myRole() == "trainer"
      && request.resource.data.gymId == myGymId()
      && exists(/databases/$(database)/documents/trainers/$(request.resource.data.trainerId))
      && get(/databases/$(database)/documents/trainers/$(request.resource.data.trainerId)).data.uid == uid()
    )
  );
  allow update: if (owner() || myRole() == "trainer") && sameGym() && incomingSameGym();
  allow delete: if owner() && sameGym();
}
```

**Note:** The catch-all rule (line 149) must not match these two collections once they have their own
match blocks — Firestore applies the most specific matching rule, so adding explicit blocks is safe
without removing the catch-all.

### Key Technical Decisions

**1. exercises field: free text string, not array**
The existing `workout_templates` module stores exercises as a plain textarea string. Phase 7 should
follow the same convention — store `exercises` as a string in both `workout_assignments` and
`workout_sessions`. This avoids JSON parsing complexity. Phase 10 (Member Workout Logging) will
introduce structured exercise arrays; Phase 7 should not pre-optimise for that.

**2. No inline modals for the trainer session form**
The existing pattern for trainer-facing forms (see `trainer-checkin.js`, `trainers.js`) is to have
the form always visible in the panel on the left side of a `.work-grid` layout. Avoid pop-up modals
for the session creation form. Use the same `.work-grid` + form-left / list-right layout.

**3. workout_sessions NOT a sub-collection**
Keep `workout_sessions` as a top-level Firestore collection (not nested under members or trainers).
The existing `services.data.list()` method only supports top-level collections. Using a
sub-collection would require adding a new query method to firebase-init.js — unnecessary complexity.

**4. Owner view: read-only, no override**
The ENGINEERING_HANDOFF.md explicitly says: "Owner cannot override trainer assignments." The owner
view in `trainers.js` is purely informational. Do not add save/edit functionality to the owner view.

**5. myTrainer.id vs profile.uid in session records**
Always store `trainerId: myTrainer.id` (the trainers roster doc id), not `profile.uid` (Firebase
auth uid). Consistent with how `trainer_attendance` records work. Same for `memberId: myMember.id`.

**6. Firestore rules: use existing cross-doc lookup pattern**
The trainer write rule should mirror the `trainer_attendance` rule (line 133–145 of firestore.rules)
— it validates that `trainerId` in the incoming doc matches a trainers doc whose `uid` is the
caller's uid. This prevents a trainer from writing sessions on behalf of another trainer.

### Risks and Landmines

**Risk 1: workout_assignments has no index yet**
Querying `workout_assignments` by `memberId` or `trainerId` against a large collection will require
a composite Firestore index if the collection grows. For Phase 7, all assignments are loaded at boot
via `list()` (gymId filter only) and filtered client-side — no new indexes needed. Note this in the
plan as a future performance concern.

**Risk 2: trainer assigned to member via `assignedTrainer` field vs. `trainerId` in assignments**
There are two ways a trainer-member relationship is expressed:
- `members.assignedTrainer` — the trainer roster doc ID assigned to a member (used for filtering)
- `workout_assignments.trainerId` — the trainer who assigned a workout

A trainer can only write sessions for members where `member.assignedTrainer === myTrainer.id`. The
Firestore rule should enforce this but cannot easily do so without a cross-doc read (the rule would
need to read the member doc). The simpler approach: enforce the relationship in the UI (only show
the trainer's own members) and use the basic trainerId-uid check in the rule. Owner watches over
any misuse.

**Risk 3: members with no `assignedTrainer` appear in trainer's list incorrectly**
If `member.assignedTrainer` is empty or null, filter must be strict equality: 
`m.assignedTrainer === myTrainer.id` (not a loose check). Members with no trainer should not appear.

**Risk 4: workout_assignments — multiple assignment records per member**
If a trainer assigns a new template, should the old assignment be updated or a new one created?
The schema includes `assignedAt` date field suggesting history is intentional. For Phase 7, the
simplest approach: always create a new record on each assignment (history preserved). The "current"
assignment is the one with the latest `assignedAt`. Filter: 
```js
const assignments = (data.workout_assignments || []).filter(a => a.memberId === member.id);
const current = assignments.sort((a, b) => b.assignedAt.localeCompare(a.assignedAt))[0];
```

**Risk 5: Firestore catch-all rule conflict**
The catch-all at line 149 (`match /{collection}/{docId}`) gives owner write, any-same-gym read.
When adding explicit match blocks for `workout_sessions` and `workout_assignments`, Firestore uses
the most permissive matching rule. Adding explicit blocks that are MORE permissive than the
catch-all (trainer write) is correct and safe.

**Risk 6: smoke-test.mjs must be updated**
The smoke test currently does not test Phase 7 modules. The test will not fail if they are omitted,
but the ENGINEERING_HANDOFF.md requires smoke test coverage before each PR. Both new modules must
be added to smoke-test.mjs with trainer and member contexts (including the "no roster doc" empty
state path).

**Risk 7: Branch state**
The ENGINEERING_HANDOFF.md says Phase 7 should be on branch `feature/trainer-workout-assignment`
off `ui/dark-mode-animations-polish` (Phase 6). The current git status shows we are on branch
`phase7` — verify this is the correct branch. The current branch already appears to contain the
Phase 7 planning docs.
