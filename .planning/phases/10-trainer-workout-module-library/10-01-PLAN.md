# Phase 10 Plan 01: Trainer Workout Module Library + Basic Member Library

## Objective

Trainers currently land on **My Members** and can write today's session or assign a
template only in the context of an already-assigned member. They need a separate
workflow to create reusable workout modules first, without selecting a client, then
assign those modules later from the client view.

The same workout module can be promoted to **Basic**, which makes it automatically
visible to all gym members as a general workout library item. This gives members
starter workouts even before a trainer writes a custom daily session for them.

## Current Problem

- `app.js` exposes the `workouts` route only to owners.
- `modules/workouts.js` is owner-oriented and does not distinguish template ownership
  or visibility.
- Trainers can assign templates from `modules/trainer-members.js`, but they cannot
  create templates from the trainer interface.
- Trainer navigation says **My Members**, but the user-facing language should be
  **My Clients** or **Assigned Clients**.
- Member-facing `modules/my-workout.js` only shows today's session, assigned template,
  and assignment history. It does not show Basic workout modules.

## Proposed Data Model Changes

Use the existing `workout_templates` collection. Add optional metadata fields:

```js
{
  id: "workout_templates_<uid>",
  gymId: "<gymId>",
  name: "Beginner Strength A",
  goal: "Muscle Gain",
  exercises: "Squat - 3 x 10\nBench press - 3 x 10",
  notes: "Keep 90 sec rest between sets.",

  // New fields
  visibility: "private" | "basic",
  createdByRole: "owner" | "trainer",
  createdByUid: "<auth uid>",
  trainerId: "<trainers doc id, if trainer-created>",
  status: "active",
  createdAt: "...",
  updatedAt: "..."
}
```

### Visibility Semantics

| Visibility | Who Can See | Purpose |
|---|---|---|
| `private` | Owner and creator trainer | Trainer drafts or internal reusable modules |
| `basic` | Owner, trainers, all same-gym members | General starter workouts for the gym |

For existing templates with no `visibility`, treat them as owner/global templates:
visible to owners/trainers for assignment, but not automatically shown in the Basic
member library until explicitly marked `basic`.

### Task 2: Extend Workouts Module for Trainer Template Creation

**File:** `modules/workouts.js`

Update the module so it supports both owner and trainer users.

Rendering rules:
- Owner sees all same-gym templates.
- Trainer sees:
  - templates created by them
  - owner/global templates
  - `basic` templates
- Trainer can create a workout template without selecting a member/client.
- Trainer-created templates are saved with:
  - `createdByRole: "trainer"`
  - `createdByUid: context.profile.uid`
  - `trainerId: context.myTrainerId`
  - `visibility: "private"` by default
  - `status: "active"`
- Owner-created templates are saved with:
  - `createdByRole: "owner"`
  - `createdByUid: context.profile.uid`
  - `visibility: "private"` or `basic` based on the form

Add form field:

```html
<label>Visibility
  <select name="visibility">
    <option value="private">Private</option>
    <option value="basic">Basic - visible to members</option>
  </select>
</label>
```

Optional UI copy:
- Use "Workout Module" wording instead of only "Template".
- Show a small badge on cards: `Private`, `Basic`, `Owner`, or `Trainer`.

**Acceptance:**
- Trainer can open Workouts and create a module without choosing a client.
- New trainer module appears in trainer's Workouts list.
- Created module has trainer ownership metadata.
- Marking visibility as `basic` makes the module eligible for member display.

### Task 3: Firestore Rules for Trainer-Created Templates

**File:** `firestore.rules`

Add an explicit `match /workout_templates/{docId}` block before the catch-all.

Rules:
- Same-gym users can read templates.
- Owner can create/update/delete same-gym templates.
- Trainer can create a template only when:
  - incoming `gymId` equals their `gymId`
  - `createdByRole == "trainer"`
  - `createdByUid == request.auth.uid`
  - `trainerId` maps to a trainer doc whose `uid == request.auth.uid`
- Trainer can update only their own trainer-created templates.
- Trainer cannot delete owner-created templates.
- Member cannot create/update/delete templates.

**Acceptance:**
- Trainer writes to `workout_templates` are allowed for own templates.
- Trainer cannot spoof another trainer's `trainerId`.
- Member write attempts are denied.
- Owner workflows remain allowed.

## Suggested Enhancements Worth Considering

These are not required for the first implementation, but I recommend considering them
because they will make the trainer workflow feel much more complete.

### 1. Draft / Published / Archived Lifecycle

Add `status: "draft" | "active" | "archived"` so trainers can prepare modules without
showing unfinished work. Basic visibility should require `status === "active"`.

**Why useful:** avoids half-written workouts appearing to members.

### 2. Owner Approval Before Basic Publishing

Instead of letting any trainer instantly publish to Basic, use:

```js
visibility: "private" | "basic_pending" | "basic"
approvedBy: "<owner uid>"
approvedAt: "..."
```

**Why useful:** if a gym has multiple trainers, the owner may want quality control
before a module becomes visible to all members.

### 3. Template Categories and Filters

Fields:

```js
difficulty: "Beginner" | "Intermediate" | "Advanced"
goal: "Weight Loss" | "Muscle Gain" | "Strength" | "Mobility"
equipment: "Bodyweight" | "Dumbbells" | "Machines" | "Mixed"
durationMinutes: 30
```

**Why useful:** members can browse Basic modules by goal instead of reading a long list.

### 4. Duplicate / Clone Module

Add a "Duplicate" action that copies an existing owner/basic module into the trainer's
private library.

**Why useful:** trainers can adapt a gym standard plan for a specific coaching style
without modifying the shared version.

### 5. Structured Exercises Later

Current templates use free-text `exercises`. Later, support structured exercise rows:

```js
exercisesStructured: [
  { name: "Bench Press", sets: 3, reps: 10, restSeconds: 90, notes: "" }
]
```

**Why useful:** unlocks exercise logging, progress comparison, and "repeat last workout".

### 6. Member Save/Favorite Basic Workouts

Members can bookmark a Basic workout into a `member_saved_workouts` collection.

**Why useful:** members can quickly return to routines they like without needing a
trainer assignment.

### 7. Trainer Usage Analytics

Show how many clients are assigned to each module and when it was last used.

**Why useful:** trainers can clean up unused modules and see which plans are popular.

## Recommended Scope for First Build

Implement core Tasks 1-6 first.

I recommend adding **Template Categories and Filters** in the same phase only if you
want the member Basic library to feel polished immediately. I would defer Owner
Approval, Structured Exercises, Favorites, and Analytics until after the core flow is
working end-to-end.
