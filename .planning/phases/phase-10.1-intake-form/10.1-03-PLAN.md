<objective>
Add three related UX improvements to the member form:

1. **Duplicate detection** — When the owner blurs the mobile or email field while adding a
   member, the form instantly checks the already-loaded members list for a duplicate and
   shows a named inline warning with an edit shortcut. Zero extra Firestore reads.

2. **WhatsApp number field** — A separate phone field for WhatsApp (defaults to the mobile
   number; independently editable).

3. **WhatsApp opt-in checkbox** — Members can consent to WhatsApp reminders. The
   `whatsappOptIn: true/false` boolean is persisted to the Firestore member document.

After this plan:
- Typing a mobile number that already belongs to another member → yellow inline warning
  appears under the mobile field showing the member's name; clicking it pre-fills the edit form
- Same for email field
- Warning does NOT block form submission (advisory only; family accounts share numbers)
- WhatsApp number field appears after mobile, auto-fills to mobile value, stays editable
- WhatsApp opt-in checkbox appears at the end of the form; `whatsappOptIn` bool saved to Firestore
- Editing a member pre-fills both new fields; checkbox reflects saved consent state
- Form reset (Clear button) clears both warnings
</objective>

### Task 3 — Duplicate detection blur listeners (members.js bind)

**File:** `modules/members.js`

After the existing `form.gender?.addEventListener("change", updateBmi)` line, add:

```js
    // ── Duplicate detection ─────────────────────────────────────────────
    const dupWarnMobile = form.querySelector('[data-dup-warn="mobile"]');
    const dupWarnEmail  = form.querySelector('[data-dup-warn="email"]');

    function showDupWarn(el, member) {
      if (!el) return;
      if (member) {
        el.textContent = `⚠ Already registered: ${member.fullName} — tap to edit`;
        el.classList.remove("hidden");
        el.dataset.editTarget = member.id;
      } else {
        el.textContent = "";
        el.classList.add("hidden");
        delete el.dataset.editTarget;
      }
    }

    form.mobile?.addEventListener("blur", () => {
      const val    = form.mobile.value.trim();
      const editId = form.elements['id']?.value || "";
      const match  = val
        ? context.data.members.find(m => m.mobile?.trim() === val && m.id !== editId)
        : null;
      showDupWarn(dupWarnMobile, match);
    });

    form.email?.addEventListener("blur", () => {
      const val    = form.email.value.trim().toLowerCase();
      const editId = form.elements['id']?.value || "";
      const match  = val
        ? context.data.members.find(m => m.email?.trim().toLowerCase() === val && m.id !== editId)
        : null;
      showDupWarn(dupWarnEmail, match);
    });

    [dupWarnMobile, dupWarnEmail].forEach(el => {
      el?.addEventListener("click", () => {
        if (!el.dataset.editTarget) return;
        const member = context.data.members.find(m => m.id === el.dataset.editTarget);
        if (!member) return;
        // Re-use existing edit logic: pre-fill form and scroll
        Object.entries(member).forEach(([key, value]) => {
          if (form.elements[key]) form.elements[key].value = value || "";
        });
        if (form.elements.whatsappOptIn) {
          form.elements.whatsappOptIn.checked = !!member.whatsappOptIn;
        }
        updateBmi();
        root.querySelector(".panel-heading h2").textContent = "Edit Member";
        form.scrollIntoView({ behavior: "smooth", block: "start" });
        showDupWarn(dupWarnMobile, null);
        showDupWarn(dupWarnEmail,  null);
      });
    });
```

**Acceptance:**
- Entering a mobile number that belongs to an existing member and tabbing away → yellow warning appears with member name
- Entering the same mobile as the member currently being edited → no warning (excluded by `m.id !== editId`)
- Clicking the warning → form pre-fills with that member's data for editing
- Entering a unique mobile → no warning shown

### Task 4 — WhatsApp number auto-sync with mobile (members.js bind)

**File:** `modules/members.js`

After the duplicate detection block from Task 3, add:

```js
    // ── WhatsApp number auto-sync ────────────────────────────────────────
    let lastMobileSync = "";
    form.mobile?.addEventListener("input", () => {
      const whatsappEl = form.elements.whatsappNumber;
      if (!whatsappEl) return;
      if (!whatsappEl.value || whatsappEl.value === lastMobileSync) {
        whatsappEl.value = form.mobile.value;
      }
      lastMobileSync = form.mobile.value;
    });
```

**Logic:** The WhatsApp field auto-mirrors mobile as long as the user has not manually changed it (i.e., it still equals the last value we pushed). Once the user edits WhatsApp independently, `whatsappEl.value !== lastMobileSync` so the auto-sync stops.

**Acceptance:**
- Typing "9876543210" in mobile → WhatsApp number auto-fills to "9876543210"
- Manually changing WhatsApp to "9000000000" → subsequent mobile changes do NOT overwrite it
- On form reset, WhatsApp clears; next mobile input re-syncs

### Task 5 — Handle whatsappOptIn in submit handler (members.js bind)

**File:** `modules/members.js`

In the submit handler, after `const payload = formData(form);`, add the boolean coercion:

```js
      const payload = formData(form);
      payload.whatsappOptIn = payload.whatsappOptIn === "true";
```

**Why:** `FormData` omits unchecked checkboxes entirely. When unchecked, `payload.whatsappOptIn` is `undefined`; `undefined === "true"` is `false`. When checked, the value attribute `"true"` is captured; `"true" === "true"` is `true`. This always stores a boolean.

**Acceptance:** Saving with checkbox checked → Firestore has `whatsappOptIn: true`. Saving with checkbox unchecked → Firestore has `whatsappOptIn: false`.
