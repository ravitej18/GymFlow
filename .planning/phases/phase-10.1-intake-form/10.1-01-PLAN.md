are simply stored on the existing `members` document.

After this plan:
- Personal enrichment fields (blood group, occupation, activity level, fitness experience, referral source) are present and saved
- Emergency contact is split into three proper fields: name, relationship, phone
- A collapsible "Health & Medical" `<details>` section captures conditions, medications, allergies, limitations
- Three additional body measurements (hip, bicep, thigh) are added to Initial Measurements
- All new fields pre-fill correctly when editing existing members (handled by existing `Object.entries` loop)
- The initial progress record written on new-member save is extended to include hip, bicep, thigh
</objective>

### Task 6 — Handle whatsappOptIn checkbox in edit pre-fill (members.js bind)

**File:** `modules/members.js`

In the `data-edit-member` click handler, the existing `Object.entries` loop sets `.value` for all named elements. A checkbox needs `.checked` set instead. After the existing loop, add:

```js
        // Checkbox: .value setter does not change .checked state
        if (form.elements.whatsappOptIn) {
          form.elements.whatsappOptIn.checked = !!member.whatsappOptIn;
        }
```

The exact insertion point — after `updateBmi()` call is also fine:
```js
        Object.entries(member).forEach(([key, value]) => {
          if (form.elements[key]) form.elements[key].value = value || "";
        });
        if (form.elements.whatsappOptIn) {
          form.elements.whatsappOptIn.checked = !!member.whatsappOptIn;
        }
        updateBmi();
```

**Note:** The `whatsappOptIn` input element is added in Plan 10.1-03. This code is safe to add now — `form.elements.whatsappOptIn` will be `null` until Plan 03 runs, and the `if` guard protects against that.

**Acceptance:** After Plan 10.1-03 is applied, editing a member with `whatsappOptIn: true` → checkbox is pre-checked.
